🎯 今天的主軸

▍共同變化
打開這 48 小時三大 AI coding 工具的更新紀錄，你會發現一件反常的事：幾乎沒有一條在講「新增了什麼能力」。7/18 的 Claude Code v2.1.214 是一次以「安全」為主題的釋出，一口氣修掉五個權限檢查繞過的漏洞；7/17 的 GitHub Copilot 程式碼審查把「防火牆預設開啟」，並讓你用一個設定檔管理它、把用量指標下放到每個 repo；7/18 的 OpenAI Codex 0.144.6 則是修 bug、把弄亂的上下文長度救回來。三家不同公司，這幾天的關鍵字高度一致：關得住、看得清、擋得掉。
▍為什麼放在一起看
單看任何一條，都只是「例行更新」；但把三條疊在一起，方向就很清楚了。過去一年這些工具在拼的是「能不能自己讀 codebase、自己開 PR、自己跑指令」——也就是放權。當放權放到一個程度，真正的瓶頸就不再是「它會不會做」，而是「它做壞事時，我攔不攔得住、事後查不查得到」。這一週，三家不約而同開始補這一課。這不是巧合，是 AI coding 從「炫技期」進入「上生產線」的必經轉折。

過去：changelog 的頭條是「支援新模型」「新增 /某某指令」「速度更快」，安全性通常擠在最後一行。
現在：Claude Code 這版把「安全修正」放在最前面——修掉 Windows PowerShell 5.1 的權限繞過、讓超過 1 萬字元的超長指令一律先問你而不是自動跑、堵住用檔案描述子重導向（file-descriptor redirect）繞過檢查的寫法。Copilot 則把「防火牆預設開啟、限制審查時的網路存取」當成頭條功能。安全性，從腳註升格成主打。

📖 今日概念｜最小權限原則（Principle of Least Privilege）與 fail-closed
一句定義：只給一個角色「完成任務所需的最小權限」，多一分都不給；而當系統遇到「看不懂、判斷不了」的狀況時，預設選擇「擋下來」而不是「放行」——後者叫 fail closed（失效時關閉）。
生活化比喻：新員工報到，門禁卡預設只開得了自己那層樓，不是直接發一把「全棟總鑰匙」。而電子門鎖設計成停電時「自動上鎖」而非「自動彈開」——寧可讓你多刷一次卡，也不要在異常時大門洞開。說回技術：Claude Code 這版把「解析不出來的長指令」從『預設自動跑』改成『預設先問你』，就是同一個道理——異常時往安全那邊倒。
工作中的例子：給 AI coding agent 的權限，別一次全開。能跑哪些指令、能連哪些外部網路、能改哪個目錄，逐項用白名單開；Copilot 把審查防火牆「預設開啟、限制網路」，就是幫你先把大門關上，再讓你按需要開窗。
常見誤解：「先全開，出事再收」比較有效率。實際上剛好相反——權限一旦放出去很難收回，而 fail-closed 的一次確認成本很低、一次意外的代價卻可能很高（刪錯檔、外洩、跑了不該跑的指令）。摩擦感是設計的一部分，不是 bug。
與今天主軸的關係：今天三家工具的「補課」，骨子裡都是這條老原則的落地——把 agent 的預設從「方便」調回「安全」，用最小權限與 fail-closed，換來「敢把鑰匙交出去」的底氣。
